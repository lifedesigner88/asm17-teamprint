## 2. Environment and Docker Build Stabilization
대표 커밋:
- `a3b4a98 chore: standardize local and docker workflows`

### What Changed
- app 로컬 env는 `apps/*/.env.example`만 추적하고, 실제 `.env`는 실행 전에 자동 생성하도록 바꿨다.
- Compose 데모용 env는 루트 [compose.env](../../compose.env) 하나로 정리했다.
- frontend / backend Dockerfile을 추가하고 BuildKit 기반 빌드 경로를 정리했다.
- Docker와 로컬이 `node_modules`, `.venv`, Vite cache를 공유하지 않도록 경로를 분리했다.
- frontend Dockerfile은 의존성 설치 레이어 캐시가 재사용되도록 `package.json` 중심으로 재구성했다.

### Why It Mattered
- 이 단계가 없으면 local과 Docker를 번갈아 실행할 때 권한 충돌과 캐시 오염이 반복된다.
- env 역할이 명확해야 "어떤 값이 어디서 들어오는지"를 설명할 수 있다.

## 5. Capture Data Model Hardening
대표 커밋:
- `8e3da73 refactor: link capture jobs to users`

### What Changed
- `capture_jobs.owner_user_id` 문자열 연결을 제거하고 `owner_id -> users.id` FK로 바꿨다.
- `User.capture_jobs`, `CaptureJob.owner` relationship을 추가했다.
- API 응답은 기존 프론트 계약을 깨지 않도록 `owner_user_id`를 계속 노출했다.
- SQLite 테스트 DB에서도 FK가 실제로 강제되도록 보강했다.

### Why It Mattered
- 기존 구조는 유저별 분리는 되지만 DB가 참조 무결성을 보장하지 못했다.
- 이후 `voice_assets`, `image_assets`, `generation_results` 같은 도메인도 같은 방식으로 확장할 수 있으려면 FK 기반 정리가 먼저 필요했다.

## 6. Capture Submit Flow and Submission Management UI
대표 커밋:
- `bf08872 feat: submit capture draft and refine ui`
- `2860908 feat: add capture submission management`

### What Changed
- capture review 단계에서 backend `POST /capture/jobs`로 실제 제출되도록 연결했다.
- draft 저장은 `sessionStorage` 대신 브라우저 메모리 기반으로 바꿨다.
- `capture` 화면은 shadcn 스타일 공용 컴포넌트를 기준으로 다시 정리했다.
- step 페이지의 반복 구조는 feature 내부 공통 shell / action 컴포넌트로 추출했다.
- 사용자가 자신의 제출물을 카드형 목록으로 보고, 상세를 확인하고, 삭제할 수 있는 화면을 추가했다.
- later admin 화면에서도 재사용할 수 있도록 `CaptureJobCard`, `CapturePayloadSummary` 같은 컴포넌트를 분리했다.

### Why It Mattered
- 이전까지는 capture가 "입력 UI 초안" 수준이었고, 실제 저장 흐름은 완성되지 않았다.
- 이 단계 이후부터는 사용자가 입력 -> 제출 -> 조회 -> 삭제까지 한 사이클을 직접 확인할 수 있게 됐다.

## Study Notes
이 날의 흐름을 다시 공부할 때는 "기능 추가"보다 "운영 기준이 어떻게 단단해졌는가"를 보는 편이 좋다.

추천 복습 질문:
1. 왜 실행 경로를 `pnpm dev`, `pnpm docker`, `docker compose up`으로 분리했는가?
2. 왜 `apps/*/.env`는 비추적하고 `compose.env`는 추적하는 구조를 택했는가?
3. 왜 smoke test를 SQLite 기반으로 먼저 붙였고, 그 한계는 무엇인가?
4. 왜 `capture_jobs`를 문자열 user id가 아니라 FK로 바꿨는가?
5. 왜 제출 목록 / 상세 UI를 admin 재사용을 염두에 두고 컴포넌트화했는가?

이 질문에 답할 수 있으면, 이 날의 커밋 흐름을 단순한 기능 추가가 아니라 "운영 가능한 템플릿 정비"로 이해한 것이다.
