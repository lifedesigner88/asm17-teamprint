## 3. Refactor — Integer PK + Signup ID Card Removal

- `auth/models.py`: dual `id`+`user_id(String)` → single `user_id: Integer` auto-increment PK
- `auth/schemas.py`, `service.py`: removed `user_id` from signup; JWT subject = `str(user.user_id)`
- `persona/models.py`, `capture/models.py`: FKs updated to `Integer`
- `signup-page.tsx`: removed 6-char ID card (generate/copy/shuffle); signup is now email + PIN only

**Why:** Client-generated IDs required users to write down a code. Integer PK is collision-free and invisible to users.

**Files:** [auth/models.py](../../apps/backend/app/features/auth/models.py) · [signup-page.tsx](../../apps/frontend/src/features/auth/pages/signup-page.tsx)

## 4. Bilingual Persona — `data_eng` / `data_kor` + DB Seed

- `persona/models.py`: `data: JSON` → `data_eng: JSON` + `data_kor: JSON (nullable)`
- `seed.py`: full EN + KO blobs; `persona_id` `"d31sf2"` → `"sejong"`; title `"세종 페르소나 · Sejong Persona"`
- `persona/router.py`: `GET /{id}` returns `{ persona_id, title, data_eng, data_kor }`
- `persona/schemas.py`: `AskRequest.lang`; ask endpoint routes to correct blob
- `persona/service.py`: `max_tokens` 512 → 1024 (answers were truncating mid-sentence)
- Frontend `loader.ts`: all hardcoded profile data removed; fetches from DB only
- Route: `/persona/demo` → `/persona/sejong`

**Why:** Hardcoded frontend profiles required a redeploy for any content change. Both blobs in one response avoids a re-fetch on language switch.

**Files:** [seed.py](../../apps/backend/app/common/seed.py) · [persona/models.py](../../apps/backend/app/features/persona/models.py) · [loader.ts](../../apps/frontend/src/features/persona/utils/loader.ts)
